# Rebuild (guidetomyself)

**Статус:** Активный legacy — в сторах, есть платящие пользователи, требует радикальной технической переработки. **Сложность:** Высокая — legacy, полное переписывание, продуктовая и дизайнерская ответственность, продуктовая суть требует глубокого погружения через заказчика. **Модель участия:** Коммерческий проект. **Сайт:** [https://guidetomyself.com](https://guidetomyself.com)

## **Суть продукта**

Мобильное приложение «от Я до Я» для саморазвития. Спрашивает пользователя про душевное самочувствие, выдаёт задания, отслеживает прогресс. Сильная дизайнерская составляющая — важно, чтобы всё выглядело красиво и удобно.

Продукт построен на уникальной авторской методологии саморазвития — это не универсальное приложение «трекер настроения», а узко-спроектированный инструмент с собственной философией. Понять его суть по коду или скриншотам не получится. Единственный способ корректно вести этот проект — **глубокое погружение через разговоры с кофаундером Андреем**, который является носителем продуктового видения и методологии.

## **Ключевой момент для менеджера**

**Не относиться к проекту как к задаче «переписать legacy».** Это продуктовая работа с авторским содержанием. Менеджер должен:

- Установить с Андреем регулярный рабочий ритм.  
- Извлечь из него продуктовую суть: что такое «я до я», какой путь проходит пользователь, какие принципы не должны быть нарушены ни при каких обстоятельствах.  
- Задокументировать это в форме, пригодной для передачи разработчикам и дизайнерам.  
- Защищать продуктовую целостность при технических решениях по переписыванию.

Менеджер без способности к глубокому диалогу с носителем продукта здесь не справится.

## **Стадия**

- Приложение опубликовано в сторах.  
- Есть платящие пользователи.  
- Проект получен в поддержку от других подрядчиков с некачественным кодом.  
- Накопившийся техдолг делает текущую архитектуру практически неподдерживаемой.  
- Требуется полное переписывание с учётом современных технологий.

## **Тех-стек (текущий, legacy)**

- Backend: Python.  
- Mobile: React Native.  
- PostgreSQL.

## **Тех-стек (целевой при переписывании)**

- Backend: возможно Rust (вместо/в дополнение к Python) — решение принимается на этапе анализа.  
- Mobile: Flutter (вместо React Native).  
- Сохранить интеграции, данные и — критично — продуктовую логику.

## **Команда и роли**

- **Андрей** — кофаундер, заказчик, носитель продуктового видения и авторской методологии. Основной источник требований для менеджера.  
- **Co-founders** — из экосистемы акселераторов, активные проводники в мир стартапов. Через них — выход в сообщество стартаперов.  
- **Алексей Резвов** — техническая часть, архитектурные решения.

**Требуется:** менеджер, ориентированный на продукт с сильной дизайнерской составляющей, способный к глубокому погружению в авторскую методологию через работу с её носителем.

## **Задачи стажёра**

### 1\. Погружение в продукт (главный этап)

- Установить регулярный контакт с Андреем (кофаундер, заказчик).  
- Пройти приложение как пользователь — почувствовать пользовательский опыт.  
- Через серию встреч с Андреем извлечь и задокументировать:  
  - Продуктовую философию и методологию.  
  - Пользовательский путь (user journey) и его ключевые точки.  
  - Non-negotiable принципы дизайна и контента.  
  - Что отличает продукт от аналогов «трекеров настроения».  
  - Историю продуктовых решений: что пробовалось, что отвергалось и почему.  
- Превратить результаты в продуктовую спецификацию, с которой смогут работать разработчики и дизайнеры.

### 2\. Аудит текущего состояния

- Проанализировать существующее приложение, его архитектуру, техдолг.  
- Понять объём данных, типы интеграций, текущую пользовательскую базу.  
- Собрать продуктовую метрику: что пользователи делают, где отваливаются, что ценят.

### 3\. План переписывания

- Принять решения по целевому стеку (Rust vs Python на бэке — с Алексеем).  
- Определить стратегию миграции: с нуля или strangler-подход.  
- Составить план с учётом того, что **продуктовая целостность не должна пострадать** при переписывании.  
- Согласовать с Андреем и Алексеем.

### 4\. Координация и запуск

- Привлечь дизайнера (критично для этого продукта).  
- Привлечь разработчиков: Flutter mobile, backend (Rust/Python).  
- Вести проект по SDD-методологии FEP.  
- Удерживать баланс между скоростью переписывания и сохранением продуктовых нюансов.

## **Ценность проекта для стажёра**

- Живой коммерческий продукт с платящими пользователями.  
- Доступ к сообществу стартапов через co-founders.  
- Возможность участвовать в полноценной миграции с нуля по AI-first методологии.  
- **Редкая возможность работать с автором-носителем идеи**, извлекать продуктовую суть через диалог — это прокачивает продуктовые навыки сильнее, чем работа по готовому ТЗ.
